# Blink.MBTI Code Standards（工程规范）

## 1. 目的与适用范围
- 本规范用于统一 Blink.MBTI 项目代码质量与协作方式。
- 适用对象：前端（Vue 3）、后端（FastAPI）、数据层（MySQL/Redis）、测试与运维脚本。
- 规范优先级：
- 法规与安全要求 > 本规范 > 个人习惯。

## 2. 技术栈与版本基线
### 2.1 前端
- Vue 3（Composition API）
- Vite
- Pinia
- TypeScript（推荐全量启用，新增模块必须启用）

### 2.2 后端
- Python 3.11+
- FastAPI
- Pydantic v2
- SQLAlchemy 2.x + Alembic
- Celery + Redis（异步任务）

### 2.3 数据与基础设施
- MySQL 8.0+
- Redis 7+
- Telegram Bot API / Stars
- OpenAI API

## 3. 通用编码原则
- C-01：可读性优先于“聪明写法”。
- C-02：单一职责，一个函数只做一件事。
- C-03：避免隐式行为，关键逻辑必须显式命名。
- C-04：禁止硬编码业务常量（价格、次数上限、超时等），统一配置化。
- C-05：新增代码必须可测试。
- C-06：错误信息要对人类友好、对机器稳定（错误码不随意改名）。

### 3.1 优雅与可扩展性原则（必须实践）
- E-01：先设计边界，再写实现。每个模块必须先定义输入、输出、错误。
- E-02：高内聚、低耦合。路由层不写业务细节，组件层不写领域规则。
- E-03：规则单点收敛。计费、重试、语言优先级只能在一个服务中定义。
- E-04：显式状态机。支付和报告流程必须用状态迁移表示，不用散落 `if/else` 拼接。
- E-05：幂等优先。外部回调类逻辑必须支持重复调用且结果一致。
- E-06：可观测性内建。关键状态变化必须有结构化日志与追踪字段。
- E-07：向变化编程。易变项（价格、阈值、语言策略）配置化，稳定项（契约、状态）固化。
- E-08：失败路径先于成功路径设计。先定义失败返回，再写主流程。
- E-09：保持“小函数 + 清晰命名”。每个函数应能在 30 秒内被读懂。
- E-10：持续小步重构。新功能合入时，顺手修复同模块明显坏味道。

### 3.2 反例与正例（项目语境）
- EX-01 反例：在 FastAPI 路由中直接查询数据库、计算免费次数、调用 GPT、写支付记录。
- EX-01 正例：路由仅负责参数校验与响应拼装，业务由 `services` 编排，数据访问由 `repositories` 负责。
- EX-02 反例：前端多个页面各自判断“免费 2 次、重试 3 次”。
- EX-02 正例：前端只读后端返回状态，规则统一由后端规则引擎裁决。
- EX-03 反例：`successful_payment` webhook 每次都直接插入一条支付记录。
- EX-03 正例：先按 `charge_id` 做幂等检查，再决定是否写入与分发任务。
- EX-04 反例：语言优先级在不同页面复制实现。
- EX-04 正例：统一封装 `resolve_user_lang()`，全项目只维护一处。

### 3.3 开发前五问（提交前自检）
- Q-01：这段代码是否只有一个明确职责？
- Q-02：如果业务规则变更，是否只需改一个模块？
- Q-03：失败时是否有稳定错误码和可读提示？
- Q-04：关键路径是否可测试、可观测、可回放？
- Q-05：这段代码 3 个月后由他人接手，能否快速理解并安全修改？

### 3.4 经典设计原则补充（必须理解并应用）
- P-01 低入侵性原则：新增功能优先通过“扩展点”接入，禁止为实现新需求大面积修改稳定模块。
- P-02 开闭原则（OCP）：模块应对扩展开放、对修改关闭。新增策略时优先新增实现类或策略函数，不改动既有调用方。
- P-03 迪米特原则（LoD）：对象只与直接朋友通信。禁止链式深层访问（例如 `a.b.c.d`）驱动业务判断。
- P-04 依赖倒置原则（DIP）：上层业务依赖抽象接口，不依赖具体实现（数据库、第三方 SDK、模型供应商）。
- P-05 接口隔离原则（ISP）：接口保持小而专，避免“大而全”服务接口导致调用方被迫依赖无关能力。
- P-06 组合优于继承：优先使用组合与策略注入复用能力，减少继承层级和隐式耦合。
- P-07 最小惊讶原则：函数行为应与命名一致，不做调用方意料之外的副作用。

### 3.5 在本项目中的落地要求（强约束）
- LP-01 低入侵性：新增支付方式或模型能力时，不允许修改核心下单/回调主流程超过一个聚合服务文件。
- LP-02 OCP：计费规则、语言策略、重试策略采用策略模式或规则注册表；新增规则仅新增策略实现并注册。
- LP-03 LoD：前端组件不得直接读取深层全局状态结构；统一通过 store getter/composable 暴露只读字段。
- LP-04 DIP：`services` 层必须依赖 `repositories` 抽象与 `provider` 抽象，不直接 import 具体 ORM session 或第三方 SDK client。
- LP-05 组合优先：GPT 生成链路由“提示词构建器 + 校验器 + 降级器”组合，不使用多层继承基类。
- LP-06 最小惊讶：`create_*` 只创建不隐式提交支付；`get_*` 不产生写操作；写操作函数名必须包含动词（create/update/mark）。

### 3.6 快速判定清单（评审必看）
- CK-01：这次需求是否通过“新增模块/策略”实现，而不是改动多个稳定模块？
- CK-02：是否出现超过两层以上的属性链访问来驱动关键业务逻辑？
- CK-03：上层业务是否可在不改业务代码前提下替换存储实现或第三方供应商？
- CK-04：接口是否只暴露调用方需要的最小能力？
- CK-05：函数命名与副作用是否一致，是否存在“读接口偷偷写入”的行为？

### 3.7 工程实践原则补充（强烈建议）
- EP-01 KISS：默认选择最简单可维护方案，拒绝不必要复杂度。
- EP-02 YAGNI：未进入已确认需求的能力不提前实现。
- EP-03 DRY + AHA：避免重复，同时避免过早抽象；模式稳定后再抽象。
- EP-04 Fail Fast, Recover Gracefully：输入与状态错误尽早失败，外部依赖失败要有降级或重试。
- EP-05 幂等优先：支付、回调、重试入口必须可重复调用且结果一致。
- EP-06 原子性优先：计费、次数扣减、状态迁移必须保证事务一致性。
- EP-07 向后兼容：API 变更优先新增，不直接破坏已有字段契约。
- EP-08 最小权限（PoLP）：服务与账号默认最小权限，按需放权。
- EP-09 可观测性优先：关键路径默认打点、日志、trace，不做“黑盒逻辑”。
- EP-10 确定性优先：核心规则避免随机副作用，关键行为可复现可回放。
- EP-11 Boy Scout Rule：每次改动顺手改善周边可读性与一致性。
- EP-12 清晰优于巧妙：优先可读、可维护写法，避免炫技表达。

### 3.8 本项目执行要求（对应 EP-01 ~ EP-12）
- ER-01：单个需求 PR 默认不引入新基础设施；确有必要需在 PR 描述中说明收益与替代方案。
- ER-02：未进入 PRD/TECH SPEC 的功能不实现；如需预留，仅允许接口占位与注释，不落业务逻辑。
- ER-03：规则复用优先“函数库 + 策略注册”，不得复制粘贴计费/语言/重试判断。
- ER-04：所有外部调用（Telegram/OpenAI/Redis）必须设置超时、重试策略和失败降级路径。
- ER-05：支付回调与重试接口必须支持 `idempotency_key` 或等价唯一键去重。
- ER-06：涉及 `free_submits_used`、`retry_count`、`charge_id` 的更新必须在事务内完成。
- ER-07：API 废弃流程需经历“标记 deprecated -> 双版本并行 -> 下线”三个阶段。
- ER-08：生产环境数据库账号禁止 DDL 权限，迁移仅通过发布流程执行。
- ER-09：关键业务事件必须记录统一字段：`trace_id`, `user_id`, `report_id`, `charge_id`, `status`。
- ER-10：测试中需可注入时间、随机数与外部客户端 mock，确保失败场景可稳定复现。
- ER-11：改动旧代码时，至少清理命名、重复逻辑或错误处理中的一项明显问题。
- ER-12：评审中若“更短写法”可读性更差，必须选择可读性更高版本。

### 3.9 评审附加检查项（新增）
- CR-01：本次改动是否真的必要，是否符合 KISS/YAGNI？
- CR-02：是否引入了不成熟抽象，后续维护成本是否上升？
- CR-03：失败路径是否与成功路径同等完善（错误码、重试、降级）？
- CR-04：关键更新是否具备事务与幂等保障？
- CR-05：接口变更是否保证向后兼容并有迁移说明？
- CR-06：是否具备足够日志与指标支持线上排障？

## 4. 目录与模块规范

## 4.1 前端目录建议
```text
frontend/
  src/
    api/            # HTTP client 与接口封装
    stores/         # Pinia 状态管理
    views/          # 页面级视图
    components/     # 可复用组件
    composables/    # 复用逻辑（useXxx）
    utils/          # 纯工具函数
    constants/      # 常量与枚举
    types/          # TS 类型定义
```

## 4.2 后端目录建议
```text
backend/
  app/
    api/            # 路由层
    services/       # 业务编排层
    repositories/   # 数据访问层
    models/         # SQLAlchemy 模型
    schemas/        # Pydantic schema
    tasks/          # Celery 任务
    core/           # 配置/日志/安全
    utils/          # 工具函数
  migrations/       # Alembic 迁移
  tests/
```

- M-01：禁止在路由函数中直接写复杂业务逻辑。
- M-02：禁止跨层直接调用（例如路由直接操作数据库会话）。

## 5. 命名规范
### 5.1 通用
- 文件名：小写 + 下划线（Python）或 kebab-case（前端组件除外）。
- 类名：PascalCase。
- 函数/变量：snake_case（Python）/ camelCase（TS）。
- 常量：UPPER_SNAKE_CASE。
- 布尔变量以 `is_`, `has_`, `can_`（Python）或 `is/has/can`（TS）开头。

### 5.2 API 字段
- HTTP JSON 字段统一 `snake_case`。
- 接口路径使用名词短语：`/api/user/init`、`/api/compat/history`。
- 版本升级通过路径或 header 明确，不做隐式破坏性变更。

### 5.3 数据库命名
- 表名：复数小写下划线（`compat_reports`）。
- 主键：`id` 或业务型主键（例如 `user_id`）。
- 时间字段统一：`created_at`、`updated_at`。
- 枚举值统一小写英文（`pending`, `done`, `failed`）。

## 6. 代码风格与格式化
### 6.1 Python
- 必须工具：`ruff` + `black` + `isort` + `mypy`。
- 行宽：100。
- 所有公开函数必须有类型注解。
- 异步函数命名无需 `async_` 前缀，但需语义清晰。

### 6.2 Vue/TS
- 必须工具：`eslint` + `prettier` + `vue-tsc`。
- 组件优先 `<script setup lang="ts">`。
- 禁止在模板中写复杂表达式；复杂逻辑放 `computed` 或 `composable`。

### 6.3 提交前门禁
- 本地必须通过：lint、type-check、unit test。
- CI 必须重复执行并阻断不通过的合并。

## 7. 业务规则落地要求（本项目关键）
- B-01：价格、免费次数、重试上限必须来自配置，不得硬编码。
- B-02：计费相关逻辑必须幂等（`charge_id` 唯一）。
- B-03：`failed` 报告重试次数必须有明确计数与上限校验。
- B-04：语言优先级逻辑必须封装成单点函数，禁止多处复制。
- B-05：每次状态变更必须写审计日志（最少包含 user_id/report_id/status）。

## 8. 数据库与迁移规范
- DB-01：任何 schema 变更必须通过 Alembic migration，不允许手改生产库。
- DB-02：迁移脚本必须可回滚（downgrade 至少可执行）。
- DB-03：新增索引需附带原因（查询路径/唯一性/幂等保障）。
- DB-04：涉及计费/支付字段的变更必须经过双人评审。
- DB-05：事务边界要明确，避免长事务。

## 9. API 设计规范
- A-01：统一响应结构：成功返回业务对象；失败返回 `error` 与 `detail`。
- A-02：错误码语义固定：
- `400` 参数错误
- `401` 未授权
- `402` 需要支付
- `403` 禁止操作（如重试超限）
- `404` 资源不存在
- `429` 频率限制
- `500` 服务内部错误
- A-03：幂等接口必须支持 `idempotency_key`（支付与重试相关接口建议必带）。
- A-04：禁止返回未脱敏敏感信息。

## 10. 日志与可观测规范
- O-01：日志统一 JSON 结构，至少包含：`timestamp`, `level`, `trace_id`, `user_id`, `event`。
- O-02：错误日志必须带 stacktrace，但敏感数据需脱敏。
- O-03：关键业务指标必须可观测：
- submit 成功率
- free->paid 转化率
- compat done/failed 比率
- webhook 成功率
- GPT 超时率
- O-04：告警阈值必须落地配置文件并版本化。

## 11. 安全规范
- S-01：生产环境必须开启 Telegram initData HMAC 校验。
- S-02：Webhook 必须做来源校验 + 重放防护。
- S-03：密钥只允许来自环境变量或密钥管理系统，不得入库/入 git。
- S-04：用户输入默认不可信，所有入参都必须校验。
- S-05：依赖升级需经过安全扫描（至少每月一次）。

## 12. 测试规范
### 12.1 测试分层
- 单元测试：纯函数、规则引擎、状态机。
- 集成测试：API + DB + Redis（可用测试容器）。
- 端到端测试：关键链路（submit、支付回调、重试、恢复）。

### 12.2 覆盖要求
- 核心业务模块行覆盖率不低于 80%。
- 计费/支付/重试模块必须有异常路径测试。

### 12.3 必测场景（v1.0）
- 免费 2 次后触发 402。
- paid submit 不受每日免费上限限制。
- `failed` 报告最多重试 3 次。
- 重复 webhook 不重复记账。
- 断网恢复 `pending_report_id` 正常。
- 语言优先级与报告语言规则正确。

## 13. Git 与分支规范
- G-01：分支命名：`feature/*`、`fix/*`、`chore/*`。
- G-02：Commit 使用 Conventional Commits：
- `feat: ...`
- `fix: ...`
- `refactor: ...`
- `test: ...`
- `docs: ...`
- G-03：禁止直接推送 `main`。
- G-04：PR 至少 1 名评审通过后合并。
- G-05：涉及支付、鉴权、数据库迁移的 PR 必须 2 人评审。

## 14. PR 与 Code Review 规范
### 14.1 PR 要求
- PR 描述必须包含：背景、改动点、风险点、测试结果、回滚方案。
- 单个 PR 建议不超过 500 行有效改动（自动生成文件除外）。

### 14.2 Review 清单
- 是否破坏产品规则（计费、重试、语言）。
- 是否存在并发一致性问题。
- 是否存在安全风险（越权、注入、密钥泄露）。
- 是否补齐测试。
- 是否可观测（日志/指标）。

## 15. 配置与环境规范
- ENV-01：所有环境变量集中定义在 `.env.example`。
- ENV-02：新增环境变量必须写明默认值与用途。
- ENV-03：配置项命名统一大写下划线。
- ENV-04：本地、测试、生产配置必须可区分，不允许复用生产密钥。

## 16. 发布与回滚规范
- R-01：发布前必须通过 CI 全量检查。
- R-02：生产发布采用灰度或分批策略。
- R-03：数据库迁移必须先在 staging 验证。
- R-04：每次发布必须有可执行回滚预案。

## 17. 文档与注释规范
- D-01：对外接口变更必须同步更新技术文档。
- D-02：复杂业务逻辑需补充注释，解释“为什么”，而不是“做了什么”。
- D-03：废弃接口必须标注替代方案与下线时间。

## 18. Definition of Done（完成定义）
- DoD-01：功能实现符合 PRD 与 TECH SPEC。
- DoD-02：代码通过 lint、type-check、test。
- DoD-03：关键路径测试覆盖已补齐。
- DoD-04：文档已更新（接口、配置、迁移说明）。
- DoD-05：监控与告警项已补齐（若涉及线上链路）。
- DoD-06：PR 已通过评审并可回滚。

## 19. 推荐工具链（可直接落地）
### 19.1 后端
- `ruff check .`
- `black .`
- `isort .`
- `mypy app`
- `pytest -q`

### 19.2 前端
- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm build`

### 19.3 Git Hooks
- 建议使用 `pre-commit`（Python）+ `husky`（前端）做本地门禁。
